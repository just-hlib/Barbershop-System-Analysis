&#x20;Огляд проектуСистема призначена для керування записом клієнтів, обліку робочого часу майстрів (барберів), контролю складських запасів косметики та фінансової звітності. Лабораторна робота №3: Побудова UML-діаграмВідповідно до вимог проектування, було реалізовано наступні діаграми:1. Діаграма класів (Class Diagram)Описано 10 класів, що відображають логічну структуру системи.Для кожного класу визначено не менше 5 атрибутів із логічними типами даних та 5 операцій із типами значень, що повертаються.Використано різні типи відношень: асоціація, агрегація, композиція та наслідування.2. Діаграми станів (Statechart Diagrams)Побудовано 3 діаграми станів (одна загальна для системи та дві для конкретних елементів).Кожна діаграма містить не менше 6 станів.Реалізовано специфікації переходів, що включають тригери, граничні умови та дії.3. Діаграми діяльності (Activity Diagrams)Створено 3 діаграми діяльності для ключових варіантів використання: бронювання, оплата та звітність.Використано понад 6 діяльностей у кожній схемі.Застосовано стани прийняття рішення (decision) та синхронізації (fork/join).4. Діаграма компонентів (Component Diagram)Відображено фізичну архітектуру системи за тришаровою моделлю (Presentation, Business, Database logic).Усі класи системи розподілені між відповідними компонентами для забезпечення модульності. Лабораторна робота №4: Патерни програмуванняДля підвищення гнучкості та надійності системи впроваджено такі паттерни проектування:Factory Method (Фабричний метод): Для створення об'єктів різних типів користувачів (Client, Barber, Admin) на основі базового абстрактного класу.Singleton (Одинак): Для гарантування наявності лише одного підключення до бази даних у межах компонента Database logic.Strategy (Стратегія): Для реалізації різних методів оплати послуг та товарів без зміни основного коду класу Invoice.
