Огляд проекту  

Ця система допомагає керувати записами клієнтів, відстежувати робочий час барберів, слідкувати за складськими запасами косметики та вести фінансову звітність.



Лабораторна робота №3: UML-діаграми  

Для проєкту я побудував кілька UML-діаграм:



1\. Діаграма класів  

Я описав десять класів, які показують логічну структуру системи. У кожному класі є як мінімум п’ять атрибутів і п’ять операцій з конкретними типами даних. В діаграмі використовуються різні типи відношень — асоціація, агрегація, композиція, і наслідування.



2\. Діаграми станів  

Я зробив три діаграми станів: одну загальну і дві для окремих елементів. В кожній — не менше шести станів. Прописані переходи, тригери, граничні умови та дії.



3\. Діаграми діяльності  

Тут три діаграми — для бронювання, оплати та звітності. У кожній схемі понад шість діяльностей. Використав рішення (decision) та синхронізацію (fork/join), щоб усе було чітко.



4\. Діаграма компонентів  

Ця діаграма показує фізичну архітектуру системи за тришаровою моделлю: Presentation, Business, Database logic. Класи поділені між компонентами — модульність збережена.



Лабораторна робота №4: Патерни  

Щоб система була гнучкою й надійною, я впровадив такі паттерни:



\- Factory Method — для зручного створення різних типів користувачів (Client, Barber, Admin) через абстрактний клас.

\- Singleton — система підтримує тільки одне підключення до бази даних у компоненті Database logic.

\- Strategy — різні способи оплати можна додати або змінити без переписування класу Invoice.
